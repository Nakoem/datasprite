# Code Review Report — #5 缓存 fail-open + 熔断

> 审查日期：2026-07-18
> 审查范围：1 个变更文件（app/core/cache.py）
> 触发场景：#16 补测时发现 #5 缓存裸调 Redis（遗留坑），本次补上 fail-open

---

## 背景

#16 审查中发现：recall_column / recall_metric / run_sql 三处裸调缓存函数，
Redis 故障时异常直接打断问数主链路。缓存应是加速器而非依赖项。

## 改动内容

`_redis_safe` 装饰器套住全部 4 个缓存函数：

1. **fail-open**：读失败返回 None（等价未命中，调用方回退真实计算）；
   写失败静默跳过，warning 带完整堆栈
2. **30 秒熔断**：首败后冷却期内直接跳过 Redis。没有它，一次问数十几个
   缓存操作挨个傻等 1-3s 超时（redis-py 内部还带重连重试），降级模式被
   拖成龟速——初版实测 150s 都跑不完，加熔断后 111s 完整收官

## 审查发现

未发现高危/中危。低危 2 项均为有界降级行为，不阻塞提交：

1. **LOW** — 冷却到期瞬间多个并发协程可能同时探测，各付 1-3s。有界（受
   并发度限制）且罕见，不处理
2. **LOW** — rate_limiter 没有同款熔断，Redis 停机期间每请求付 ~1s
   wait_for。有界、已 fail-open，记入改进清单（后续可抽公共熔断器复用）
